# ALVA Design System

> **PoC:** Figma as the single source of truth — design tokens and component contracts read directly from the Figma API and translated into a production-ready React component library.

[![Storybook](https://img.shields.io/badge/Storybook-Live_Demo-FF4785?logo=storybook&logoColor=white)](https://leoalvarenga-design.github.io/alva-design-system/)

---

## Concept

Traditional design-to-code workflows require manual translation: a designer specs something in Figma, a developer interprets it. Drift is inevitable.

This project replaces that step with a **direct API connection**:

```
Figma Variables (tokens)  ──▶  alva.css  (CSS Custom Properties)
Figma Component Set       ──▶  alva-ui   (React + TypeScript)
alva-ui                   ──▶  alva-storybook  (living documentation)
                          ──▶  alva-web         (Next.js demo app)
```

Every component was generated by reading the Figma file via the **Model Context Protocol (MCP)** — variant axes, token bindings, layer structure, interaction states — then implementing it in code with zero manual colour-picking or spacing guesswork.

## Token Architecture

Three Figma variable collections map to three CSS scopes:

```css
/* primitives — theme-agnostic raw values */
:root {
  --primitives-color-pink-500: #e91e8c;
  --primitives-spacing-16: 16px;
  /* ... */
}

/* semantic — aliases with meaning, light/dark modes */
:root, [data-theme="light"] {
  --semantic-surface-action-primary-default: var(--primitives-color-pink-500);
}
[data-theme="dark"] {
  --semantic-surface-action-primary-default: var(--primitives-color-pink-400);
}

/* responsive — typography & spacing scale per breakpoint */
@media (min-width: 768px) { ... }
```

Components consume **only semantic tokens** — they never reference a primitive or a hardcoded value directly. This means dark mode works for every component automatically, with no JavaScript involved.

## Components

| Component | Figma source | Key props |
|---|---|---|
| `Button` | `Button` component set | `variant` (Primary/Secondary/Tertiary), `label`, `showIconLeft/Right`, `iconLeftGlyph/Right`, `disabled`, `fullWidth` |
| `Icon` | `Icon` component set | `size` (16/20/24), `glyph` (ReactNode) |
| `Label` | `Label` component set | `labelText`, `required`, `optional` |
| `Field` | `Field` component set | `status` (Default/Error/Success/Disabled), `placeholderText`, icon slots |
| `Input` | `Input` component set | All Field props + `showLabel`, `showHelper`, `helperText`, `labelText` |
| `Card` | `Card` component | `title`, `body`, `showCTA`, `cta` (ReactNode), `image` (ReactNode) |

All interactive states (hover, pressed, focus, disabled) are implemented via CSS pseudo-classes — no JavaScript state management in the components.

## Key Technical Decisions

**No hardcoded values anywhere.**  
Every colour, spacing, border-radius and font is a CSS Custom Property referencing a token. Changing a primitive value in `alva.css` cascades to every component.

**Icon colour via `currentColor`.**  
Icon SVGs use `fill="currentColor"` or `stroke="currentColor"`. The parent component sets `color` via a semantic icon token — glyphs are swap-safe and never need their own token binding. This mirrors Figma's mask + `_color` rectangle pattern.

**State forcing for galleries.**  
Hover/pressed/focus states are simulated in Storybook and the demo app using `data-force-hover` / `data-force-pressed` / `data-force-focus` attributes on wrapper elements, with CSS overrides that mirror the real pseudo-class rules. No components were modified.

**Figma-to-code traceability.**  
Each component file documents the Figma node ID, collection IDs and variable IDs used. Token comments reference the exact `VariableID` from Figma, making it auditable.
